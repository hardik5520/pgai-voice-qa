# Architecture

The system is built around a live audio bridge rather than a classic speech to text, LLM, text to speech pipeline. `server.py` opens two WebSocket connections for the duration of each call, one to Twilio's Media Streams (carrying the actual phone call audio in both directions) and one to OpenAI's Realtime API (a voice to voice model, not a text based one). Audio arriving from the phone call is forwarded straight into the Realtime API as it arrives, and audio the model generates is forwarded straight back into the call, with no intermediate transcription or text step in that loop. This choice was deliberate: the assessment's top priority is a natural, well paced conversation with sensible turn taking, and a voice to voice model handles its own turn detection and interruption timing internally, whereas chaining separate speech to text, LLM, and text to speech services adds latency and pushes turn taking logic onto us. That same bridge also handles interruptions explicitly, when the caller starts talking while hardik-bot's reply is still playing, the in flight response is truncated and Twilio's playback buffer is cleared, so the two sides don't talk over each other.

`make_call.py` places each outbound call through Twilio and tells it which scenario to run via a URL path, which `personas.py` maps to a system prompt describing who the caller is and what they're trying to accomplish that call. Rather than a fixed script, the model improvises the actual conversation from that prompt, reacting to whatever pretty-bot says. All eleven scenarios use the same caller identity (Hardik Arora, born May 5th, 2000), since Pretty Good AI's test account ties a persistent patient record to the calling phone number, meaning state genuinely carries over between calls the same way it would for a real returning patient. `reschedule_appointment` and `cancel_appointment` are written to look up whatever's currently on file rather than assuming a specific date, since an earlier version that hardcoded a date broke the moment that appointment got rescheduled by a different call. `run_all_scenarios.sh` runs the full set of scenarios unattended, placing each call, waiting for it to finish, transcribing it, and moving to the next, with `cancel_appointment` deliberately interleaved before every scenario that books a new appointment so each one starts from a clean slate instead of repeatedly hitting an existing appointment conflict.

Recording and transcription are handled entirely separately from the live call, and by a different model. Twilio records each call in dual channel mode, one physical channel per speaker, which makes accurate transcription possible without any diarization guesswork. `transcribe.py` downloads that recording, uses ffmpeg to split it into its two channels and to produce a normal, listenable mixed down copy (a raw two channel file has each voice fully on one side, which sounds broken on ordinary playback), then runs silence detection on each channel to find where that speaker actually starts and stops talking. Each detected burst is transcribed in isolation by OpenAI's transcription model rather than transcribing the whole file at once, since a transcription model's own timestamps drift on long, mostly silent, single speaker audio, especially right after a pause, whereas timestamps measured directly from the audio are exact. The results from both channels are merged by that measured time into one chronological, speaker labeled transcript.
